# task: pr-hunter — PR/Issue確認タスク(設計段階)

目的: open状態のPRやIssueを確認し、対応が必要なものを整理して人間に報告する。

**現状はまだ設計メモです。GitHub API / GitHub CLIへの接続は必須にしません。**
接続を有効にするのは、人間が承認し、下記のTODOを解決してからです。
接続前でも、ローカルのgit情報(ブランチ、ログ)から分かる範囲の報告は行えます。

## 将来の想定フロー(dry-run設計)

1. open PRの確認(想定)
   - open状態のPR一覧を取得し、放置日数・レビュー状況を整理する
2. open Issueの確認(想定)
   - open状態のIssue一覧を取得し、優先度・古さ・対応可能性を整理する
3. CI失敗の確認(想定)
   - 直近のCI実行結果を確認し、失敗しているPR・ブランチを特定する
4. レビューコメントの確認(想定)
   - 未返信のレビューコメントを抽出し、対応が必要なものを列挙する
5. 修正案の作成
   - 失敗や指摘に対する修正案を、変更理由・影響範囲つきでまとめる
   - rubrics/code.md の基準で自己評価する
6. 人間承認待ちで停止
   - 修正案をレシートに記録し、**実行せずに止まる**
   - push / PR作成 / merge / コメント投稿は人間の承認後のみ
