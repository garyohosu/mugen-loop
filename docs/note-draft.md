# 私は何もしないループエンジニアになりたい。Claude Codeで「mugen-loop」を作る実験

<!-- note記事の下書き。公開は人間が行う。このファイルはnote-experimentタスクの追記対象。 -->

## はじめに

AIに仕事を頼むようになってから、気づいたことがあります。
私は一日に何回も「続きをお願い」と打っている、ということです。

Claude CodeのようなAIエージェントは、頼めばコードを書き、ドキュメントを整え、
調査までしてくれます。それなのに、私は毎回チャット欄の前に座って、
指示を出し、結果を確認し、次の指示を考えています。

これは本当に「AIに任せている」と言えるのでしょうか。
AIが作業者になった代わりに、私が「AIの管理という作業」の作業者になっただけではないか。

そこで実験を始めることにしました。テーマはこうです。

**私は何もしない。ループが働く。**

## 何もしないループエンジニアとは

先に誤解を解いておくと、「何もしない」はサボることではありません。

何もしないループエンジニアとは、**作業が自動で回る構造を設計する人**のことです。
自分の手で反復作業をしない代わりに、次のことをします。

- 目的を決める
- AIが越えてはいけない境界線を決める
- 成果の良し悪しを測る評価基準を決める
- 危ないときに止まる条件を決める
- そして、ログを見る

作業はしないけれど、責任は放棄しない。
責任を「毎回の指示」ではなく「設計」という形に変換する。
それがこの実験で目指す役割です。

## すでに私はバーチャルカンパニーを動かしている

実は、私にとって「何もしないループ」は、完全な空想の産物ではありません。すでに現実の運用の延長線上にあります。

私はこれまで、AIエージェントを活用した複数の「バーチャルカンパニー（仮想企業）」を立ち上げ、実際に運用してきました。
（ポートフォリオ：https://garyohosu.github.io/portfolio/ ）

そこでは、AIが自動でトレンドを追って記事を書き、別のAIが翻訳し、さらに別のAIがブログを更新・公開し、エラーや不正がないかを監視してログを残す、といった無人運用実験が日々動いています。Auto AI Blog、AITecBlog、AI-Broker、AozoraDailyTranslations、WorldClassicsJP、Writer、そして監視を行うCompanyGuardianなど、さまざまな役割を持ったAIたちが「会社」として稼働しているのです。

しかし、これまでは個別プロジェクトごとにその都度AIの動き方を設計し、個別に動かしていました。
このやり方では、プロジェクトが増えるたびに管理がバラバラになり、再現性や安全性の担保が難しくなっていきます。

`mugen-loop` は、これらの無人AI運用をゼロから作り直すためのものではありません。むしろ、すでに動き出している無人AI運用を、より安全に、より説明可能に、より再現しやすく整理し直すための「共通テンプレート」として設計されました。

これまで個別に定義していたAIの動作を、`CONTRACT`（契約）、`schedule`（スケジュール）、`rubrics`（評価基準）、`state`（状態保存）、`receipts`（監査ログ）という共通の構造に当てはめることで、無人AI運用の標準化を目指しています。

## mugen-loop はバーチャルカンパニーのOSになるかもしれない

複数のバーチャルカンパニーを運用する中で見えてきた課題は、「全体のガバナンス」です。

記事を書くAI、翻訳するAI、レビューするAI、そしてそれを監視するAI。それぞれが個別かつ孤立して動いているだけでは、全体の安全性や、何か問題が起きたときの説明可能性（なぜその判断をしたのか）がどうしても弱くなってしまいます。

もし、すべてのAIエージェントが `mugen-loop` という共通の枠組みの上で動くとしたらどうでしょうか。

すべてのAIに共通の行動契約（CONTRACT）を与え、稼働予定（schedule）を整理し、同一の自己点検基準（rubrics）で品質を担保し、実行するたびに同一フォーマットの監査ログ（receipts）を残す。
こうすることで、`mugen-loop` は、個別のAIプロジェクトを束ね、安全に制御するための「運用OS」のような役割を担えるのではないかと考えています。

もちろん、現時点ではまだ壮大な実験段階です。いきなりすべての権限をOSに委ねるのではなく、まずは「安全な見回り・報告・記録・レビュー」の範囲から着実に実績を積み上げていこうとしています。

## mugen-loop の考え方

実験用に作ったリポジトリが `mugen-loop` です。
チャットの指示は消えますが、リポジトリに置いたファイルは消えません。
そこで、AIに毎回説明していたことを、全部ファイルにして先に置いておきます。

- **CONTRACT(契約)** — AIに何を許可し、何を禁止し、いつ止まるべきかを書いた文書。AIは作業前に必ずこれを読む
- **schedule(スケジュール)** — いつ、どのタスクを動かすかの設計メモ。「毎朝9時に見回り」のように書いておく
- **rubrics(評価基準)** — AIが自分の成果を自己点検するための物差し。コード用、文章用、安全用の3種類
- **state(状態保存)** — 前回どこまでやったか、次に何を引き継ぐかを記録するファイル。AIの「記憶」の代わり
- **receipts(監査ログ)** — 実行のたびにAIが残す「レシート」。何をしたか、なぜしたか、何をしなかったかを書く

つまり、「AIに毎回続きをお願いする」のではなく、
**AIが読むべき契約と評価基準を先に置いておく**、という発想の転換です。

## 最初は報告だけにする

正直に言うと、いきなり「AIが勝手にコードを直してpushするループ」を作ることもできます。
でも、それはやりません。危ないからです。

AIループの事故は、悪意からではなく「良かれと思って」から起きるはずです。
勝手にpushする、勝手にファイルを消す、勝手に依存関係を上げる。どれも善意で起こりえます。

だからmugen-loopの最初の目標は「勝手に修正するAI」ではなく、
**「勝手に見回り、判断材料を整理し、ログを残すAI」**です。

段階はこう考えています。

1. 報告だけ — AIは見回って、レシートを残す。変更はしない
2. 提案 — AIは改善案を作る。実行は人間が承認してから
3. 承認付き自動化 — 実績のあるタスクだけ、承認フローつきで実行を許可する

信頼は実績からしか生まれない。だからまず、報告の実績を積んでもらいます。

## 実験1: フォルダ構造を作る

最初の実験は、この仕組みをフォルダ構造として作ることでした。
Claude Codeに設計を渡して、次の構成を作ってもらいました。

```
mugen-loop/
├── README.md                  … プロジェクトの説明
├── CLAUDE.md                  … Claude Codeへの永続指示
├── CONTRACT.md                … AIループの契約
├── contract.local.example.md  … 個人設定の例(実物はgit管理外)
├── docs/
│   └── note-draft.md          … この記事の下書き
├── .claude/loops/
│   ├── settings.json          … 安全設定(dry-run、push禁止など)
│   ├── schedule.yml           … スケジュール設計メモ
│   ├── rubrics/               … 評価基準(code / writing / safety)
│   ├── tasks/                 … タスク定義(daily-check / pr-hunter / blog-review / note-experiment)
│   ├── state/checkpoint.json  … 状態保存
│   └── receipts/              … 監査ログ置き場
└── scripts/                   … 見回りとレシート作成のスクリプト
```

タスクは今のところ4つです。daily-check は状態確認と見回り。
pr-hunter は PR / Issue / CI 確認の将来タスク(まだ設計メモ段階)。
blog-review は記事本文やREADMEなど文章そのもののレビュー。
note-experiment は実験ログをnote記事に育てるタスク(この記事もそれで育てています)。

面白かったのは、CLAUDE.mdに安全ルールを書いておくと、
Claude Code自身がそのルールを引用しながら「これは承認が必要なのでやりません」と
止まるようになることです。ルールをチャットで言うのではなく、
リポジトリに置くだけで、AIの振る舞いの初期値が変わります。

## 実験2: receiptsを残す

もう一つの柱が「レシート」です。
AIが実行するたびに、`receipts/日付/` フォルダにMarkdownでログを残します。

書く内容は3つに決めました。

- **何をしたか** — 実行した確認、作った提案
- **なぜしたか** — どの契約・どの基準に基づいた判断か
- **何をしなかったか** — 見送った操作と、その理由

特に大事なのは3つ目だと思っています。
「何をしなかったか」が残っていると、AIが境界線の手前で止まったことを後から確認できます。
止まった記録は、失敗の記録ではなく、契約どおりに動いた証拠です。

人間の仕事は、このレシートを読むことです。
レシートがたまってくれば、「このタスクは任せて大丈夫そうだ」という判断材料になります。

## 実験3: Codexのリミットは思ったより早く来る

Phase 1.5(マルチエージェント化)を試すなかで、Codex execをレビュー係に使ってみました。
便利なのですが、常用のレビュー係にすると利用枠を思ったより早く使い切ってしまいます。

しかも、別プロジェクトの修理作業にもCodexを使っていると、mugen-loop側で使える枠が
さらに減ります。「困ったらCodexに聞けばいい」という運用は、枠が有限である以上長続きしません。

そこで、Codexは常用エージェントではなく、**高価なセカンドオピニオン**として扱うことにしました。
普段のレビューや、Codexの利用枠に達したときはAntigravityやClaude Code自身のセルフレビューに
逃がし、Codexは重要レビュー・公開前レビュー・危険操作前レビューのときだけ呼び出します。

これも「何もしないループエンジニアリング」の一部だと思っています。
道具を増やすことよりも、道具をいつ・どれだけ使うかを設計することの方が、
ループを長く保つためには大事だと気づきました。

## 実験5: mugen-loopに自分自身を報告させる

ここまで作った CONTRACT、receipts、checkpoint、QandA、todo を、人間が毎回全部読むのは
正直しんどくなってきました。ファイルは増えるほど「見に行く」コストが上がります。

そこで、mugen-loop自身に状態レポートを作らせることにしました。
`scripts/generate-loop-status.py` を実行すると、これらのファイルを読み取って
`docs/loop-status.md` という1枚のレポートにまとめてくれます。

これは自動修正ではありません。ファイルを1つ書き出すだけで、それ以外には触れません。
観察と報告だけなので、いきなり修正やpushをするAIループではなく、
最初に決めた Phase 1(Report Loop)の範囲にきちんと収まっています。

「何もしないループエンジニア」にとって、まず必要なのは自動作業よりも状態が見えることだ、
と作りながら気づきました。何を任せるかを考える前に、今どうなっているかが一目で分かる場所が
要るのです。`docs/loop-status.md` が、その最初のダッシュボードになりました。

## 今後やりたいこと

まだ実験は始まったばかりで、今は手動でタスクを起動しています。
今後試したいことを挙げておきます。

- Claude Codeのスケジュール実行やcron、GitHub Actionsと接続して、schedule.yml どおりに自動起動する
- GitHub CLIと接続して、open PRやIssueの見回り(pr-hunter)を実データで動かす。まずは読み取り専用で
- ブログ下書きのレビューなど、見回り対象を増やす
- receiptsを集計して「ループの働きぶりレポート」を作る
- 実績のあるタスクから、承認付きで実行権限を少しずつ渡していく

うまくいくかは分かりません。分からないからこその実験です。
失敗したら、それもこの記事の続きに書きます。

## まとめ

mugen-loopで確かめたいことは、突き詰めると一つです。

人間が作業するのではなく、**ループを設計する側に回れるか。**

AIに毎回指示を出す暮らしから、
契約と評価基準と監査ログを設計して、あとはログを眺める暮らしへ。

「私は何もしない。ループが働く。」

そう言える日を目指して、まずは毎朝の見回りループから育てていきます。
興味を持った方は、リポジトリの構成(CONTRACT、rubrics、receipts)だけでも
真似してみてください。チャットへの指示が一つ減るだけで、設計者の感覚が少し味わえます。
