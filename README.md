# mugen-loop

**私は何もしない。ループが働く。**

## このリポジトリの目的

mugen-loopは、AIに毎回チャットで指示するのではなく、
契約、スケジュール、評価基準、状態保存、監査ログによって、
AIが自律的に仕事を進める仕組みを実験するためのリポジトリです。

ただし、初期段階では危険な自動化は行いません。
最初の目標は「勝手に修正するAI」ではなく、
**「勝手に見回り、判断材料を整理し、ログを残すAI」**です。

## 何もしないループエンジニアとは

「何もしない」は、サボることではありません。
反復作業を自分の手でやらない、ということです。

AIに毎回「続きをお願い」と言うのは、実はまだ作業をしています。
指示を出すという作業、確認するという作業、思い出させるという作業です。

何もしないループエンジニアは、その代わりにループを設計します。

人間がやること:

- 目的を決める
- 境界線を決める
- 評価基準を決める
- 停止条件を決める
- ログを見る

AIがやること:

- 契約を読む
- 予定に従って確認する
- 評価基準で自己点検する
- 状態を保存する
- 監査ログを残す
- 危険な操作の前で止まる

人間は、作業者から設計者になる。
AIは、チャット相手からループ実行者になる。
これがmugen-loopの中心思想です。

## なぜチャットではなくフォルダ構造でAIを動かすのか

チャットの指示は消えます。セッションが変われば、AIは前回の約束を忘れます。
だから毎回同じ説明を繰り返すことになり、人間が「AIの記憶係」という作業者になってしまいます。

フォルダ構造に置いたファイルは消えません。

「AIに毎回続きをお願いする」のではなく、
**AIが読むべき契約と評価基準を先に置いておく。**

AIは起動するたびに同じ契約を読み、同じ基準で自己点検し、同じ場所にログを残します。
人間は作業をしない。ただし、責任を放棄するわけではない。
契約と境界線という形で、責任を設計に変換するのです。

## 各ファイルの役割

- `CONTRACT.md` — 契約。AIに何を許可し、何を禁止し、いつ止まるべきかを定めます
- `.claude/loops/schedule.yml` — スケジュール。いつ、どのタスクを動かすかの設計メモです
- `.claude/loops/rubrics/` — 評価基準。AIが自分の成果を自己点検するための物差しです(code / writing / safety)
- `.claude/loops/state/checkpoint.json` — 状態保存。前回どこまでやったか、次に何を引き継ぐかを記録します
- `.claude/loops/receipts/` — 監査ログ。実行のたびに「何をしたか、なぜしたか、何をしなかったか」のレシートを残します
- `.claude/loops/tasks/` — タスク定義。daily-check(見回り)、pr-hunter(PR/Issue確認の設計)、note-experiment(mugen-loop実験のnote記事化)、blog-review(文章レビュー)

記事関連の2タスクは役割が違います。

- blog-review は「文章そのもの」を見る係。既存の記事下書きやREADME、docs/note-draft.md の本文について、タイトル、H1、読みやすさ、誤字脱字、参考リンク、公開前TODOを確認します
- note-experiment は「実験ログ」を見る係。receipts、dream.md、firststep.md、checkpoint.json を読んで、mugen-loop実験そのものをnote記事に育てます

たとえるなら、blog-review は記事を読む係、note-experiment は実験を記事に育てる係です。
- `.claude/loops/settings.json` — 安全設定。dry-run、push禁止などの初期値です
- `docs/note-draft.md` — note記事「私は何もしないループエンジニアになりたい」の下書きです
- `scripts/` — 見回りとレシート作成のための安全なスクリプトです

## 最初は「報告だけ」から始める理由

信頼は実績からしか生まれません。
AIの報告が正確かどうか分からない段階で、AIに修正権限を渡すのは順番が逆です。

mugen-loopは3段階で育てます。

1. **報告だけ** — AIは見回り、確認し、レシートを残す。変更はしない
2. **提案** — AIは改善案を作る。実行は人間が承認してから
3. **承認付き自動化** — 実績のあるタスクだけ、承認フローつきで実行を許可する

receipts/ にたまったログを読み返し、AIの判断が信頼できると確認できてから次の段階に進みます。

## 危険な自動化を避ける理由

AIループの事故は、悪意からではなく「良かれと思って」から起きます。
勝手にpushする、勝手に消す、勝手に依存関係を上げる。どれも善意で起こりえます。

だからmugen-loopでは、初期状態で以下をすべて禁止しています。

- 人間の承認なしの git push / Pull Request作成 / merge
- 人間の承認なしのファイル削除
- 人間の承認なしの依存関係更新
- 本番環境への反映
- 秘密情報(APIキー、トークン、パスワード)の表示・記録

安全装置を外すのは人間だけ。外すときは一つずつ、ログを確認しながら。

## Claude Codeでの使い方

このリポジトリをClaude Codeで開くと、CLAUDE.mdが自動的に読み込まれます。
まずはこう頼んでみてください。

```
daily-checkタスクを実行して
```

Claude Codeは `.claude/loops/tasks/daily-check.md` の手順に従って見回りを行い、
結果を報告し、`.claude/loops/receipts/` にレシートを残します。
ファイルの変更は行いません。提案までです。

夜にはこう頼めます。

```
note-experimentタスクを実行して
```

その日のレシートを読み、note記事に使える発見を `docs/note-draft.md` への追記案として整理します。

## note記事化するための実験ログの残し方

このリポジトリ自体が記事の素材です。実験ログは次の3層で残します。

- `receipts/` — 一次ログ。実行のたびに「何をしたか、なぜしたか、何をしなかったか」を記録する
- `state/checkpoint.json` — 要約された現在地。フェーズと引き継ぎ事項
- `docs/note-draft.md` — 記事の下書き。note-experimentタスクがreceiptsから発見を抜き出して追記案を作る

失敗や不明点も正直に記録します。うまくいった話だけの実験ログは、実験ログではありません。

## 既存のバーチャルカンパニー運用との関係

mugen-loopは、作者がすでに運用している[AIエージェントによるバーチャルカンパニー群](https://garyohosu.github.io/portfolio/)（Auto AI Blog、AITecBlog、AI-Broker、AozoraDailyTranslations、WorldClassicsJP、Writer など）を安全に運用するための共通テンプレートにもなり得ます。

個別のAIプロジェクトに `CONTRACT` / `schedule` / `rubrics` / `state` / `receipts` を持たせることで、ブラックボックス化しがちな無人運用を「説明可能」にし、安全に制御するための「運用OS」として機能させることを目指しています。

将来的には、既存の監視プロジェクトである `CompanyGuardian` などとも連携し、異常検知時にmugen-loop側の契約（CONTRACT）に沿って安全に停止させるような仕組みも視野に入れています。

## 今後の拡張案

- Claude Codeのスケジュール実行やcron、GitHub Actionsから schedule.yml を読んで定期起動する
- GitHub CLI (`gh`) と接続し、pr-hunterを実データで動かす(読み取り専用から始める)
- Issueの整理やブログ下書きレビューなど、タスクの種類を増やす
- receiptsを集計して「ループの働きぶりレポート」を自動生成する
- checkpoint.json に学びを蓄積し、次回実行に引き継ぐ
- 通知連携(ただし送信内容は人間が事前確認する)

どの拡張も、原則は変わりません。
**人間はループを設計する。AIはループを回す。危険な操作の前では必ず止まる。**
